# ТЗ на создание сайта оценки анализа звонков менеджеров продаж

## 1. Цель и результат
Создать веб-сервис для контроля качества звонков отдела продаж, который:
- оценивает звонки по этапам скрипта продаж;
- показывает сводные показатели и динамику на дашборде;
- выявляет и отображает риски срыва сделок;
- позволяет настраивать критерии оценки и хранить версии изменений.

Ожидаемый результат MVP: рабочий внутренний продукт для РОПа/тимлидов с единым контуром "оценка звонка -> риск -> контроль по менеджерам".

## 2. Роли и права доступа
- `admin` (РОП/владелец): полный доступ, включая настройки и пользователей.
- `lead` (тимлид): доступ к дашборду, рискам, менеджерам, звонкам.
- `manager` (опционально в MVP): доступ только к своим звонкам и оценкам.

## 3. Информационная архитектура (разделы сайта)
- `Дашборд`
- `Риски`
- `Менеджеры`
- `Настройки` (вкладки: `Быстрая настройка`, `Основные`, `Квалификация`, `Продажи`)

Глобальные фильтры:
- период (`Месяц`, `Неделя`, `Вчера`, `Сегодня`, произвольный диапазон);
- менеджер (`Все менеджеры` + конкретный менеджер);
- уровень оценки;
- версия шаблона продаж;
- переключатель `Показать с риском`.

## 4. Функциональные требования по экранам

### 4.1 Дашборд
Блоки:
1. KPI-карточки:
   - `Всего звонков`
   - `Риски`
   - `Общая оценка`
2. `Качество обработки (средние показатели)`:
   - `Приветствие`
   - `Выявление потребностей`
   - `Усиление боли`
   - `Презентация`
   - `Отработка возражений`
   - `Закрытие (след. шаг)`
3. `Динамика (сутки)`:
   - серия `Звонков`
   - серия `Риски срыва (шт)`
   - серия `% Рисков срыва`
4. Таблица `Звонков (Продажи)`:
   - `Дата / Менеджер`
   - `Сделка`
   - `Оценка`
   - этапные колонки: `Привет.`, `Потреб.`, `Боль`, `Презент.`, `Возр.`, `Закрыт.`
   - `Результат / Риск` (причина риска/комментарий)

Поведение:
- сортировка по дате, оценке, риску;
- фильтрация от глобальных фильтров;
- клик по строке открывает карточку звонка.

### 4.2 Риски
- список звонков с риском и причиной;
- фильтры по периоду, менеджеру, типу риска, уровню риска;
- статусы обработки риска: `новый`, `в работе`, `закрыт`;
- переход в карточку звонка.

### 4.3 Менеджеры
- рейтинг менеджеров по средней оценке;
- метрики: количество звонков, средняя оценка, `% звонков с риском`;
- разрез по этапам (сильные/слабые стороны).

### 4.4 Настройки -> Продажи
Поля:
- главная цель менеджера на этапе продаж (финальное действие воронки);
- товары и услуги компании;
- корректные следующие шаги успешного звонка;
- условия фиксации риска срыва;
- УТП компании;
- истинные возражения;
- ложные возражения.

### 4.5 Настройки -> Критерии оценки этапов
Для каждого этапа хранить описание уровней:
- `0-20 (Провалено)`
- `21-40 (Слабо)`
- `41-60 (Средне)`
- `61-80 (Хорошо)`
- `81-100 (Эталон)`

Этапы:
- Приветствие
- Выявление потребностей
- Усиление боли
- Презентация
- Отработка возражений
- Закрытие сделки (назначение следующего шага)

## 5. Карточка звонка
Состав:
- метаданные звонка (дата, менеджер, сделка, длительность, источник);
- итоговая оценка;
- оценки по каждому этапу;
- текстовые обоснования (почему выставлен балл);
- риск и причина риска;
- транскрипт (если доступен);
- версия критериев, по которой считался звонок.

## 6. Источник данных и интеграции (решение для MVP)
Принятое решение:
1. Основной источник в MVP: ручная загрузка CSV/JSON через UI.
2. Дополнительный канал в MVP: REST API для пакетной загрузки звонков из внешней CRM/телефонии.
3. Интеграции с CRM/телефонией "в реальном времени" не входят в MVP.

Требования к импорту:
- валидация формата;
- лог ошибок по строкам;
- статус обработки (`в очереди`, `обработан`, `ошибка`).

## 7. Модель оценки и правила риска (утверждено для MVP)

### 7.1 Формула оценки
- Каждому из 6 этапов присваивается балл `0..100`.
- Итоговая оценка звонка = взвешенная сумма этапов:
  - Приветствие: `10%`
  - Выявление потребностей: `20%`
  - Усиление боли: `15%`
  - Презентация: `20%`
  - Отработка возражений: `20%`
  - Закрытие сделки: `15%`
- Итог округляется до целого.

### 7.2 Правила риска срыва
Риск = `true`, если выполняется хотя бы одно условие:
- оценка этапа `Закрытие сделки` < 41;
- нет зафиксированного следующего шага (CTA отсутствует);
- этап `Отработка возражений` < 41 при наличии возражений в звонке;
- итоговая оценка звонка < 50;
- найдено нарушение, явно указанное в настройках риска.

Уровень риска:
- `высокий`: 2+ триггера;
- `средний`: 1 триггер;
- `низкий`: нет триггеров, но итог 50-60.

## 8. Версионность критериев
- Любое изменение критериев создает новую `CriteriaVersion`.
- Новые оценки считаются только на актуальной версии.
- Исторические оценки не пересчитываются автоматически.
- Для пересчета требуется отдельный явный запуск администратором.

## 9. Сущности данных (MVP)
- `User(id, role, name, email, isActive, createdAt)`
- `Manager(id, userId, team, isActive)`
- `Call(id, managerId, dealId, startedAt, durationSec, source, transcriptUrl, recordingUrl)`
- `CallScore(id, callId, criteriaVersionId, totalScore, stageScoresJson, riskLevel, riskReason, createdAt)`
- `Risk(id, callId, type, level, reason, status, assigneeId, createdAt, updatedAt)`
- `SalesSettings(id, goal, products, nextSteps, riskRules, usp, trueObjections, falseObjections, updatedBy, updatedAt)`
- `StageCriteria(id, criteriaVersionId, stageName, scoreRange, description)`
- `CriteriaVersion(id, versionName, comment, createdBy, createdAt, isActive)`
- `ImportJob(id, sourceType, fileName, status, totalRows, errorRows, createdBy, createdAt)`

## 11. Нефункциональные требования
- Формат: web, desktop-first (основной брейкпоинт 1366+), рабочий tablet-view.
- Безопасность: аутентификация + RBAC + аудит изменений настроек.
- Производительность: время ответа API дашборда до 2 сек на выборке до 10 000 звонков.
- Доступность: uptime 99.5% для внутреннего контура.
- Логи: централизованные application/error/audit логи.
- Экспорт: выгрузка таблиц в CSV (XLSX как расширение).

## 12. Технологический стек (рекомендуемый и зафиксированный)
- Frontend: `React + TypeScript + Vite`, UI-kit `Ant Design` или `MUI`.
- Backend: `Node.js (NestJS) + TypeScript`.
- DB: `PostgreSQL`.
- Кэш и очереди: `Redis` (для импортов и фоновых задач).
- Хранение файлов: `S3-совместимое` object storage.
- Инфраструктура: `Docker Compose` (MVP), далее переход на `Kubernetes` при масштабировании.
- CI/CD: GitHub Actions/GitLab CI (lint, test, build, deploy).

## 13. Границы MVP
Входит:
- 4 основных раздела (`Дашборд`, `Риски`, `Менеджеры`, `Настройки`);
- импорт звонков (CSV/JSON + API);
- оценка по 6 этапам и итоговый балл;
- риск-правила и статусы обработки риска;
- карточка звонка.

Не входит:
- авторасшифровка аудио;
- real-time интеграции с телефонией;
- AI-рекомендации следующей реплики менеджеру;
- multi-tenant архитектура.

## 14. Критерии приемки
- Фильтры периода/менеджера синхронно обновляют все блоки дашборда.
- Для каждого звонка отображаются итоговый и этапные баллы.
- Для звонков с риском отображается причина и уровень риска.
- Изменение критериев создает новую версию и применяется к новым оценкам.
- Разделы `Дашборд`, `Риски`, `Менеджеры`, `Настройки` доступны согласно ролям.
- Импорт минимум 10 000 строк завершается без падения сервиса, ошибки строк логируются.

## 15. Этапы реализации
1. Проектирование БД и API контрактов.
2. Реализация авторизации и ролей.
3. Реализация экранов дашборда и таблицы звонков.
4. Реализация рисков и карточки звонка.
5. Реализация настроек критериев и версионности.
6. Импорт данных и журнал обработок.
7. Тестирование, нагрузочная проверка, выпуск MVP.
