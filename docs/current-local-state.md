## Навигация

Меню разделено по частоте использования:

| Группа | Разделы |
|--------|---------|
| **работа** | План закупки *(стартовый экран)* |
| **справочники** | Словарь, Спецификации |
| **конфигурация** | Прайсы, Настройки |
| **быстрый заказ** | Заказ, Результат — прежний сценарий с ИИ, отдельно от плана |

При загрузке нового прайса **удаляются только строки `prices`** этого поставщика. Привязки SKU в Словаре **восстанавливаются автоматически** по названию строки в файле (точное или нормализованное совпадение). Если строки в новом прайсе нет — под полем загрузки показывается **список продуктов**: каноническое имя и «в прайсе искали: …».

## Экраны

### Прайсы

- Список **всех** поставщиков из БД (изначально 2 из seed, можно **добавлять новых**).
- Карточка: имя, минимальный заказ (RUB), сохранение отдельно от прайса.
- Загрузка `.xls` / `.xlsx` (колонки A название, B единица, C цена) — **заменяет** старый прайс; результат **сразу под полем файла** (не вверху страницы).
- Авто-восстановление привязок SKU; при расхождении — список продуктов для правки в Словаре.
- **Требования к файлам** — свёрнутый блок внизу страницы (`<details>`).
- Блок **«Новый поставщик»** — `POST /api/suppliers`.

### Спецификации *(variant B, этап 3)*

- Выбор канонического продукта.
- CRUD правил `product_specs` с областями: глобально, отдел, локация, локация+отдел, поставщик и комбинации.
- Поля: текст для поставщика, даты действия, флаг «добавлять в заказ».
- Превью: какой комментарий уйдёт поставщику при выбранных локации/отделе/поставщике (по приоритету scope).

### Словарь *(variant B, этап 2)*

- CRUD канонических продуктов (`name`, `default_unit`, `category`, active).
- **Изменить** — редактирование названия в деталях продукта (`PUT /api/products/{id}`).
- **Удаление продукта** — с подтверждением и отчётом о связях (`GET delete-impact`, `DELETE`).
- Привязка SKU: выбор поставщика → **«Найти в прайсе»** (локально + ИИ, `POST /api/products/suggest-skus`) → кандидаты или ручной ввод.
- Статус «нет в прайсе», если строка исчезла из файла поставщика.

### План закупки *(variant B, этапы 4.1–4.5)*

Wizard из пяти шагов:

**1. Спрос** — **«Новый план»**: название (по умолчанию `Заказ ДД.ММ.ГГГГ`), ответственный **Женя** / **Андрей**; в списке планов имя вида `Заказ … · Женя (черновик)` — статус в скобках обновляется по этапам. Ввод спроса по **локация × отдел**, переразобрать весь план.

**2. Проверка** — сопоставление со словарём: канонический продукт и SKU у всех поставщиков:

**3. Распределение** — 3 режима:

- **Оптимизация с учетом минимального заказа** *(по умолчанию)* — MILP, запасной greedy при отсутствии решения;
- **Только минимальная цена по позиции** — базовый «самый дешевый» сценарий без учета минималок;
- **Гибрид** — старт как «мин. цена» + блок подсказок «как дотянуть до минимального заказа» (какие позиции выгоднее переназначить).

Один поставщик на продукт; ручная смена поставщика сохраняется.

**4. Заказы поставщикам** — сборка `supplier_order_lines` из распределения + specs; превью по поставщик×локация×отдел; правка комментария; **Скачать все xlsx** (лист на каждую связку).

Дополнительно:
- блок **«Сообщения поставщикам»** с готовым текстом по каждому поставщику;
- формат: `Локация + Отдел` -> строки `товар, количество, спецификация`;
- кнопка «Скопировать».

**5. Сводка** — внутренний обзор: продукт × поставщики (цена / кол-во / сумма), фильтры локация и отдел, итоги и мин. заказы, проблемные строки без продукта или allocation; **Скачать сводный xlsx**.

Детали проверки:
- статусы: OK / **частично** (SKU есть хотя бы у одного поставщика) / нужен продукт / нет SKU / не разобрано;
- фильтр «Только проблемные» по умолчанию **выключен**;
- **локально:** точное имя, подстрока, Жаккар по словам + формы (*яблоко* ≈ *яблоки*);
- **ИИ (YandexGPT):** для строк, которые локально не сопоставились — если настроен `backend/.env`;
- **автоподсказки** похожих продуктов из словаря на каждой проблемной строке;
- блок **«Нет в словаре»** — без локального и ИИ-матча → **«Добавить в словарь»**;
- кнопка «Поиск продукта в словаре» — локально + ИИ (при наличии ключей).

### Заказ *(быстрый заказ, старый MVP)*

- Текстовый список: `Название количество единица` (`кг`, `г`, `гр`, `л`, `мл`).
- **Проверить ввод** — парсинг, превью таблицы, пересчёт `г/гр → кг`, `л → мл`, колонка «Пересчёт».
- Поддерживаются форматы:
  - без единицы (`Картофель 7` -> `7 кг`);
  - слитно (`5кг`);
  - с дефисом перед количеством (`редис - 4кг`);
  - с упаковками (`Руккола 5 пачек по 0125` -> `0.625 кг`);
  - записи с ведущим нулём как десятичные (`03` -> `0.3 кг`, `025` -> `0.25 кг`).
- **Сопоставить** — долгий запрос к `/api/match` (ИИ + fallback).
- Текст заказа и превью **не сбрасываются** при переходе на «Результат» и обратно.
- **Очистить** — новый список с нуля.

### Настройки *(variant B, этап 1)*

- **Локации** — CRUD: код (`loc_1`…), название, порядок сортировки, флаг «активна». По умолчанию 4 точки из сида.
- **Отделы** — только просмотр: `kitchen` (Кухня), `bar` (Бар).

### Результат

- Таблица после сопоставления: продукт, единица, кол-во к заказу.
- Для **каждого поставщика**: колонки **Цена** и **Кол-во** (число колонок растёт с числом поставщиков).
- Строка «Итого», подсветка позиций без матча, комментарии по строкам.
- Статус: с ИИ / без ИИ (degraded mode).
- **Скачать xlsx** — кнопка **вверху** страницы; в Excel — цена, кол-во, сумма по каждому поставщику.

## Что хранится в БД

| Таблица | Содержимое |
|---------|------------|
| `suppliers` | поставщики, мин. заказ, `last_price_upload_at` |
| `prices` | строки прайсов (название в прайсе, ед., цена) |
| `settings` | folder_id, model_name для YandexGPT |
| `locations` | точки закупки (code, name, sort_order, is_active) |
| `departments` | отделы кухня / бар (фиксированный справочник) |
| `canonical_products` | словарь канонических продуктов |
| `supplier_skus` | связи канон. продукта с позициями прайсов поставщиков |
| `product_specs` | комментарии/требования к продукту по scope |
| `procurement_batches` | план закупки: `title`, `plan_label`, `responsible`, `status` |
| `demand_lines` | строки спроса по локации и отделу |
| `allocations` | назначение поставщика на каждую строку спроса |
| `supplier_order_totals` | суммы по поставщикам и прохождение мин. заказа |
| `schema_migrations` | применённые SQL-миграции |

**Не хранится:** история заказов, канонический каталог продуктов, версии комментариев к поставщикам (следующие этапы variant B).

## Сопоставление (как работает сейчас)

1. Парсер разбирает текст заказа.
2. Backend отправляет заказ и прайсы в **YandexGPT** (если настроен `.env`).
3. При ошибке/таймауте/битом JSON — **fallback без ИИ** (поиск по БД, score, мин. цена).
4. Распределение количеств и сумм; комментарий в результате — **на лету** (не отдельный справочник specs).

Оптимизация «глобально дешевле с учётом мин. заказа» и **запрет дробить продукт между поставщиками** — **ещё не реализованы** (см. variant B).
